# nanobot 检查点（Checkpoint）管理机制分析

> 日期：2026-06-30
> 话题：`/stop` 中断恢复、检查点数据结构、消息持久化路径、多轮工具调用丢失问题

---

## 1. 整体定位

nanobot 的检查点是一个**纯内存运行时检查点**机制，核心目的：当 agent 任务被 `/stop` 中断时，保存半成品上下文，让用户的下一条消息能在完整的对话历史基础上继续。

- **不是**文件级别快照（不是 VM snapshot）
- **没有**用户可见的检查点管理命令或 API 端点
- 存储在 `session.metadata["runtime_checkpoint"]` 中，随 session JSONL 文件持久化

---

## 2. 数据模型

检查点 payload 结构（`loop.py:1702-1705`）：

```python
{
    "phase": "awaiting_tools" | "tools_completed" | "final_response",
    "iteration": int,              # 当前 LLM 迭代轮次
    "model": str,                  # 使用的模型名称
    "assistant_message": {...},    # assistant 角色的消息 dict（仅最新一轮）
    "completed_tool_results": [...],  # 已完成的 tool 结果列表（仅最新一轮）
    "pending_tool_calls": [...],   # 已发出但未执行的 tool call 列表
}
```

三个 phase 的含义：
- **`awaiting_tools`**：LLM 返回了 tool calls，还没开始执行
- **`tools_completed`**：所有 tool 已执行完毕，等待下一轮 LLM 调用
- **`final_response`**：LLM 给出最终回复（无 tool calls），turn 完成

## 3. 核心生命周期

### 发射（Emit）—— `AgentRunner._emit_checkpoint()` (`runner.py:1298-1305`)

```python
async def _emit_checkpoint(self, spec, payload):
    callback = spec.checkpoint_callback
    if callback is not None:
        await callback(payload)
```

Checkpoint 回调在 `AgentLoop.dispatch_turn()` 中注入（`loop.py:726-729`），每次发射都会调用 `_set_runtime_checkpoint()`，**立即** `sessions.save()` 持久化。

### 保存（Save）—— `AgentLoop._set_runtime_checkpoint()` (`loop.py:1702-1705`)

```python
def _set_runtime_checkpoint(self, session, payload):
    session.metadata[self._RUNTIME_CHECKPOINT_KEY] = payload
    self.sessions.save(session)
```

### 恢复（Restore）—— `AgentLoop._restore_runtime_checkpoint()` (`loop.py:1729-1781`)

1. 从 `session.metadata["runtime_checkpoint"]` 读取
2. 构建 `restored_messages` 列表：
   - assistant 消息
   - 已完成的 tool result 消息
   - 未完成的 tool call → 替换为 `"Error: Task interrupted before this tool finished."`
3. 去重后追加到 `session.messages`
4. 清除检查点和 `pending_user_turn` 标记

### 清除（Clear）—— `AgentLoop._clear_runtime_checkpoint()` (`loop.py:1713-1715`)

```python
session.metadata.pop(self._RUNTIME_CHECKPOINT_KEY, None)
```

## 4. 完整数据流

### 正常路径

```
runner._run_core()
  │  while True 循环内累积 messages 列表
  └─ return messages → dispatch_turn()
                          │
                          └─ ctx.all_messages = all_msgs  (loop.py:1505)
                               │
                               ▼
                          _state_save()  (loop.py:1511)
                               │
                               ├─ _save_turn(session, ctx.all_messages, ...)
                               │     │
                               │     ├─ 过滤空 assistant、孤儿 tool、截断过长结果
                               │     ├─ 清理运行时注入的 context tag
                               │     └─ session.messages.append(entry)  (loop.py:1666)
                               │
                               ├─ _clear_runtime_checkpoint(session)  (loop.py:1546)
                               └─ sessions.save(session)              (loop.py:1547)
                                    │
                                    └─ JSONL 原子写入 (manager.py:588-599)
```

### `/stop` 中断路径

```
/stop 命令
  │
  ▼
asyncio.CancelledError 抛出
  │
  ▼
_dispatch except 块  (loop.py:1028-1056)
  ├─ _restore_runtime_checkpoint(session)  ← checkpoint → session.messages
  ├─ _clear_pending_user_turn(session)
  └─ sessions.save(session)               ← 持久化
  │
  ▼
raise（任务终止）

... 下一条消息到来 ...

  │
  ▼
_state_restore()  (loop.py:1383)
  └─ _restore_runtime_checkpoint(session)  ← 兜底（去重保证幂等）
```

## 5. 去重机制

### 消息指纹 `_checkpoint_message_key()` (`loop.py:1718-1727`)

```python
def _checkpoint_message_key(message):
    return (
        message.get("role"),
        message.get("content"),
        message.get("tool_call_id"),
        message.get("name"),
        message.get("tool_calls"),
        message.get("reasoning_content"),
        message.get("thinking_blocks"),
    )
```

7 个语义字段，**刻意排除 `timestamp`**（恢复时 timestamp 被重设为当前时间）。

### 尾部重叠检测 (`loop.py:1766-1777`)

```python
overlap = 0
max_overlap = min(len(session.messages), len(restored_messages))
for size in range(max_overlap, 0, -1):        # 从大到小尝试
    existing = session.messages[-size:]         # session 尾部
    restored = restored_messages[:size]         # 检查点头部
    if all(key(left) == key(right) for left, right in zip(existing, restored)):
        overlap = size
        break
session.messages.extend(restored_messages[overlap:])
```

**从最大可能重叠开始向下试探**，找到最大匹配窗口后只追加不重叠的部分。保证两次恢复入口（`/stop` 时 + 下条消息时）不会重复追加。

## 6. 消息持久化到 Session

[`SessionManager.save()`](nanobot/session/manager.py#L574-L621)：

- **格式**：JSONL，第一行 metadata，后续每行一条消息
- **路径**：`{workspace}/sessions/{safe_key}.jsonl`
- **原子性**：先写 `.jsonl.tmp`，再 `os.replace()` 原子 rename
- **fsync**：默认关闭（OS page-cache 足够），graceful shutdown 时开启
- **文件内容示例**：
  ```jsonl
  {"_type":"metadata","key":"...","metadata":{...},"last_consolidated":...}
  {"role":"user","content":"帮我查个东西","timestamp":"..."}
  {"role":"assistant","content":"好的...","tool_calls":[...],"timestamp":"..."}
  {"role":"tool","tool_call_id":"xxx","name":"shell","content":"...","timestamp":"..."}
  ```

### 正常路径的消息写入

`_save_turn()` (`loop.py:1607-1666`) 在写入每条消息前做了关键处理：

1. **跳过空 assistant 消息**（无 content 且无 tool_calls）
2. **丢弃孤儿 tool 结果**（tool_call_id 未在已有 assistant 消息中声明过的）
3. **截断过长 tool 结果**（超过 `max_tool_result_chars`）
4. **清理运行时 context tag**（user 消息中的 `_RUNTIME_CONTEXT_TAG`）

## 7. 关键缺陷：多轮工具调用中 `/stop` 丢失历史

### 问题场景

```
Round 1: LLM → tool_a, tool_b → 全部执行完毕 ✅
Round 2: LLM → tool_c → /stop 中断 ❌
```

### 根因

检查点每次发射**覆盖**同一个 key，且每次只存当前轮的状态：

| 时机 | `assistant_message` | `completed_tool_results` | `pending_tool_calls` |
|------|---------------------|--------------------------|----------------------|
| Round 1 `tools_completed` | assistant_msg_1 | [tool_a_result, tool_b_result] | [] |
| Round 2 `awaiting_tools` | **assistant_msg_2** | **[]** ← 硬编码空数组 | [tool_c] |

Round 1 的内容被完全覆盖。而 runner 内部完整的 `messages` 列表（包含所有轮次）随 `CancelledError` 抛出而随栈帧销毁——`ctx.all_messages = all_msgs`（`loop.py:1505`）永远不会执行。

### 恢复结果

```
session.messages 中只恢复了：
  assistant_msg_2           ← "我想调用 tool_c 来..."
  tool_c → error placeholder ← "Error: Task interrupted before this tool finished."

Round 1 的 assistant_msg_1、tool_a_result、tool_b_result → 全部丢失
```

### 可能的修复方向

**方案 A**：让 checkpoint 累积而非覆盖。维护跨越迭代的 `all_completed` 列表，checkpoint 中存所有轮次的 assistant_messages + tool results。

**方案 B**：直接把 runner 的 `messages` 列表存入 checkpoint（最简单，但 checkpoint 体量变大）。

**方案 C**：在 `awaiting_tools` 阶段不覆盖 `completed_tool_results`，改为继承上一个 checkpoint 的已累积结果。

## 9. 关键设计点总结

| 方面 | 现状 |
|------|------|
| **持久化时机** | 每次 checkpoint 发射立即 `sessions.save()`，crash-safe |
| **去重保障** | 基于 7 字段指纹 + 尾部重叠检测，幂等安全 |
| **Fork 隔离** | `runtime_checkpoint` 在 `_FORK_VOLATILE_METADATA_KEYS` 中，fork 时剥离 |
| **子 Agent** | 子 agent 只更新 status 展示，不写父 session |
| **用户可见性** | 无 CLI 命令、无 API 端点，完全内部机制 |
| **多轮覆盖问题** | ⚠️ 检查点只存最后一轮，多轮工具调用的中间结果在 `/stop` 时会丢失 |
| **自动过期** | 无过期机制，未消费的检查点永久保留在 metadata 中 |
