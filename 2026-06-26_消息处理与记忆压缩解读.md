## 1. `sender_id` 是什么

**位置**：`nanobot/bus/events.py` — `InboundMessage.sender_id`

**含义**：发送这条消息的用户标识，由各 channel 从平台 API 提取（Telegram user id、Discord user id、飞书 open_id 等）。

### 与 `chat_id` 的区别

| 字段 | 含义 | 示例 |
|------|------|------|
| `sender_id` | 发送者（用户） | 某用户的 platform id |
| `chat_id` | 会话/频道 | 群 ID、DM 线程 ID |

群聊中多人共用同一个 `chat_id`，但 `sender_id` 不同。

**默认 session key** 是 `channel:chat_id`，**不是**按 `sender_id` 划分：

```python
@property
def session_key(self) -> str:
    return self.session_key_override or f"{self.channel}:{self.chat_id}"
```

### 主要用途

1. **Channel 层权限与配对** — `is_allowed(sender_id)`，未授权则拒绝或发 pairing code（`channels/base.py`）
2. **注入 LLM 运行时上下文** — `ContextBuilder._build_runtime_context` 输出 `Sender ID: xxx`
3. **区分消息来源类型** — 特殊 sentinel 值：
   - `"subagent"` → 子 agent 回调
   - `"cron"` → 定时任务
   - `"runtime"` / `"system:continuation"` / `"webui-settings"` → 内部控制
   - `"user"` → CLI / gateway 直连
4. **日志与历史溯源**

## 2. 消息入队前的四步准备（`loop.py` 1150–1162）

一条消息进入 LLM 调用之前，`AgentLoop` 依次做以下处理（`_process_system_message` 与普通用户消息的 `_state_restore` / `_state_compact` 逻辑相同）：

```text
崩溃恢复 → Auto Compact → Token 压缩（含 Replay 预归档）
```

### 2.1 崩溃 / 中断恢复（1150–1153）

| 方法 | 场景 | 行为 |
|------|------|------|
| `_restore_runtime_checkpoint` | turn 跑了一半（有 assistant、tool 进度） | 从 `metadata["runtime_checkpoint"]` 写回 history；未完成 tool 填占位错误 |
| `_restore_pending_user_turn` | 只存了 user、还没生成回复 | 补一条 `"Error: Task interrupted before a response was generated."` |

**触发时机**：新消息到来、开新 turn 之前。也覆盖 `/stop`、`gateway 重启` 等中断，不仅是进程崩溃。

**正常完成的 turn** 在 `_state_save` 里会 `clear` 两个标记，不会触发 restore。

### 2.2 Auto Compact — 空闲 TTL 压缩（1155）

```python
session, pending = self.auto_compact.prepare_session(session, key)
```

- **触发条件**：会话**空闲超过 TTL**（`session_ttl_minutes`），后台 `_archive` 已压缩
- **行为**：reload session；取出压缩摘要 `pending`
- **摘要去向**：`build_messages(session_summary=pending)` → system prompt 的 `[Archived Context Summary]`
- **物理保留**：约 **8 条**最近消息（`AutoCompact._RECENT_SUFFIX_MESSAGES = 8`）

### 2.3 Token 预算压缩（1159–1162）

```python
await self.consolidator.maybe_consolidate_by_tokens(
    session,
    replay_max_messages=self._max_messages,
)
```

**Dream Consolidator**，与 Auto Compact 是两套机制。

#### 预算计算

```python
budget = context_window_tokens - max_completion_tokens - SAFETY_BUFFER
target = budget × consolidation_ratio   # 默认 ratio = 0.5
```

#### 流程

1. **Replay 预归档**（见 2.4）— 即使 token 未超也先执行
2. **估算** `estimate_session_prompt_tokens(session)`
3. **`estimated < budget`** → 不压，直接返回
4. **`estimated >= budget`** → 最多 5 轮，逐段 `archive()` 摘要，推进 `last_consolidated`

### 2.4 Replay 窗口预归档（`memory.py:894–897`）

```python
last_summary = await self._consolidate_replay_overflow(
    session,
    replay_max_messages,
)
```

**核心逻辑**：未合并消息超过 replay 上限时，先把**窗口外**的旧消息摘要归档，**与 token 是否超标无关**。

| 项目 | 值 |
|------|-----|
| replay 上限 | `_max_messages`，**默认 120**（不是 8） |
| 归档方式 | LLM 摘要 → `history.jsonl`；推进 `last_consolidated` |
| 物理删除 | **不删**；旧消息仍在 `session.messages`，只是 `get_history` 不再 replay |

`get_history` 从 `messages[last_consolidated:]` 起，再按 `max_messages` 截尾：

```python
unconsolidated = self.messages[self.last_consolidated:]
start_idx = recent_message_start_index(unconsolidated, max_messages, extend_to_user=...)
```

## 3. 三种压缩机制对比

| 机制 | 代码入口 | 触发条件 | 保留规模 | 摘要注入 |
|------|----------|----------|----------|----------|
| **崩溃恢复** | `_restore_runtime_checkpoint` / `_restore_pending_user_turn` | 上一轮被打断 | 修 history，非压缩 | 无 |
| **Auto Compact** | `auto_compact.prepare_session` | 会话空闲 TTL | 约 **8 条** | `[Archived Context Summary]` |
| **Replay 预归档** | `_consolidate_replay_overflow` | 未合并消息 > **120** | replay 窗口内 | `_last_summary` metadata |
| **Token 压缩** | `maybe_consolidate_by_tokens` | prompt token ≥ budget | 压到 target 以下 | `_last_summary` metadata |

## 5. 一句话总结

**`sender_id` 标识谁发的，不参与默认 session 划分；消息进 LLM 前先修中断残局，再按「空闲 TTL → replay 条数 → token 预算」三层递进压缩历史，避免丢上下文又控制成本。**
